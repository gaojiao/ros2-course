# 6.服务

前面我们学习了ROS2中基于`话题`的通信方式，这种方式适用于`单向`、`持续`的消息传递，例如传感器数据的发布与接收。但在机器人系统中，还存在许多需要`请求-响应`模式的交互场景：比如查询机器人当前位置、控制机械臂执行特定动作并等待完成确认、修改参数后获取是否成功的反馈等。这些场景需要一种`双向的`、`同步的`通信机制，而ROS2中的`服务（Service）`正是为满足此类需求设计的。

本章将从`服务`的基本概念入手，介绍服务通信的特点与核心要素，然后通过Python代码实现自定义的服务端与客户端节点。

## 6.1 服务的基本概念

### 6.1.1 服务通信模型

`服务`是 ROS2 中另一种核心通信机制，采用`请求-响应`（Request-Response）模式，其通信模型包含两个核心角色：

+ `服务端（Server）`：提供特定功能的节点，等待客户端的请求并进行处理，最终返回响应结果。
+ `客户端（Client）`：向服务端发送请求的节点，等待服务端处理完成后接收响应。

通信流程如下：

1. 服务端启动后，注册一个`服务名称`并等待请求；
2. 客户端启动后，通过`服务名称`查找服务端并建立连接；
3. 客户端发送`请求数据（Request）`给服务端；
4. 服务端接收请求并处理，生成`响应数据（Response）`返回给客户端；
5. 客户端接收响应，完成一次服务调用。

这种模式与话题通信的主要区别在于：

+ 话题是`单向`、`异步`的（发布者只管发送，不关心是否被接收）；服务是`双向`、`同步`的（客户端发送请求后需等待响应）。

+ 话题适用于持续数据传输（如传感器流、状态流数据）；服务适用于按需触发的操作（如单次任务执行）。

### 6.1.2 服务名称

与话题类似，服务也通过唯一的名称进行标识，例如turtlesim中的`/set_pen`（设置画笔）、`/spawn`（生成新海龟）等。客户端必须通过正确的服务名称才能找到对应的服务端。
例如，先启动海龟仿真节点：

```
ros2 run turtlesim turtlesim_node
```

可使用`ros2 service list`命令可查看当前系统中活跃的服务：

```
$ ros2 service list
/clear
/kill
/reset
/spawn
/turtle1/set_pen
/turtle1/teleport_absolute
/turtle1/teleport_relative
/turtlesim/describe_parameters
/turtlesim/get_parameter_types
/turtlesim/get_parameters
/turtlesim/list_parameters
/turtlesim/set_parameters
/turtlesim/set_parameters_atomically
```

### 6.1.3 服务类型（Service Type）

知道了`服务名称`后，使用`ros2 service type <服务名称>`命令可查看该服务的类型，如：

```
$ ros2 service type /clear
std_srvs/srv/Empty
```

服务的数据格式由`服务类型`定义，包含请求数据结构和响应数据结构两部分。ROS2 中服务类型的文件以`.srv`为后缀，其语法格式为：

```
# 请求部分（Request）
数据类型1 成员名1
数据类型2 成员名2
...
---
# 响应部分（Response）
数据类型3 成员名3
数据类型4 成员名4
...
```

其中```---```是请求与响应的分隔符。

例如，ROS2 内置的`std_srvs/srv/SetBool`服务类型定义如下：

```
bool data  # 请求：设置的布尔值
---
bool success  # 响应：操作是否成功
string message  # 响应：附加信息字符串
```

### 6.2 服务端与客户端开发

**步骤 1：创建功能包​**

在工作空间（如~/ros2_ws）的src目录下，创建名为`my_srv`的包，依赖库为`rclpy`和`example_interfaces`。

```
cd ~/ros2_ws/src
ros2 pkg create --build-type ament_python my_srv --dependencies rclpy example_interfaces
```

+ `example_interfaces`功能包中包含现成的`AddTwoInts`服务类型，即计算两个数的和。在此我们直接使用，省去自定义.srv文件环节。该服务文件名为`AddTwoInts.srv`，内容为：
    ```
    int64 a
    int64 b
    ---
    int64 sum
    ```

**步骤 2：编写服务端代码​**

在my_srv/my_srv/下创建`my_server.py`文件，内容如下：

```python
# 1. 导入核心库
import rclpy
from rclpy.node import Node
from example_interfaces.srv import AddTwoInts

# 2. 定义服务端节点类
class AddServer(Node):
    def __init__(self):
        super().__init__("add_server")  # 节点名
        # 创建服务：类型、服务名、回调函数
        self.srv = self.create_service(
            AddTwoInts, "add_two_ints", self.calc_sum
        )

    # 3. 定义请求处理逻辑
    def calc_sum(self, request, response):
        response.sum = request.a + request.b  # 计算
        self.get_logger().info(f"收到请求：{request.a} + {request.b}")
        return response

# 4. 启动节点
def main():
    rclpy.init()
    server = AddServer()
    rclpy.spin(server)  # 持续等待请求
    rclpy.shutdown()

if __name__ == "__main__":
    main()
```


**步骤 3：编写客户端代码​**

在my_srv/my_srv/下创建`my_client.py`文件，内容如下：

```python
import rclpy
from rclpy.node import Node
from example_interfaces.srv import AddTwoInts

class AddClient(Node):
    def __init__(self):
        super().__init__("add_client")
        # 创建客户端
        self.client = self.create_client(AddTwoInts, "add_two_ints")
        # 等待服务端启动
        while not self.client.wait_for_service(timeout_sec=1.0):
            self.get_logger().info("等待服务端...")

    # 发送请求
    def send_request(self, a, b):
        req = AddTwoInts.Request()
        req.a = a
        req.b = b
        # 异步获取响应
        self.future = self.client.call_async(req)

def main():
    rclpy.init()
    client = AddClient()
    client.send_request(3, 5)  # 自定义参数
    # 等待响应结果
    while rclpy.ok():
        rclpy.spin_once(client)
        if client.future.done():
            res = client.future.result()
            client.get_logger().info(f"结果：{res.sum}")
            break
    rclpy.shutdown()

if __name__ == "__main__":
    main()
```

**步骤 4：配置入口点**

在setup.py文件的entry_points入口参数中添加：

```python
entry_points={
    'console_scripts': [
        "server_node = my_srv.my_server:main",
        "client_node = my_srv.my_client:main",
    ],
},
```

**步骤 5：构建运行**

先构建：

```
cd ~/ros2_ws
colcon build
```

构建完成后，启动服务端：

```
ros2 run my_srv server_node
```

服务端启动后，打开新的终端，查看可用服务：

```
ros2 service list  # 应显示/add_two_ints
```

此时，启动客户端：

```
ros2 run my_srv client_node
```

客户端会向服务端发送两个数的求和请求，服务端接收到请求数据后进行计算，将运算结果返回至客户端。

或者使用`ros2 service call`命令直接调用服务测试：

```
ros2 service call /add_two_ints example_interfaces/srv/AddTwoInts "{a: 4, b: 6}"
```

## 6.3 动作（Action）

目前，我们学习了ROS中的两种通信方式，即`话题`和`服务`。

+ **话题**（Topics）：单向异步通信方式，基于**发布-订阅**(Publishe-Subscribe）模式工作，适用于传感器数据流、速度指令等**持续传输**场景。

+ **服务**（Services）：双向同步通信方式，基于**请求-响应**(Request-Response）模式工作，适用于查询机器人位置、控制灯的开关等**即时应答**场景。

有些情况下，还需要第三种通信方式，即`动作`。

+ **动作**（Actions）：双向异步通信方式，基于**目标-反馈-结果(支持取消)**（Goal-Feedback-Result）模式工作，适用于机器人导航、机械臂抓取等**长时间、需反馈、可中断**的任务场景。

相比服务的**请求-响应**单次交互，**动作**（Actions）支持：

+ 实时反馈（如机械臂运动进度、路径规划百分比）
+ 任务取消（中途停止正在执行的操作）
+ 状态查询（查看任务是否在运行 / 完成 / 失败）

因此，服务适合短时间同步任务，如"计算两个数的和"；动作适合长时间异步任务，如"让机器人移动到5米外"，需实时知道移动距离，也能随时叫停。

关于动作（Action）的开发及使用方法可参考[ROS2 Action](https://docs.ros.org/en/humble/Tutorials/Intermediate/Writing-an-Action-Server-Client/Py.html)相关文档，此处不再介绍。
